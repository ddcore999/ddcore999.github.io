# **1.컴퓨터 과학 기초**

## **진법이란?**

: 한 자릿수에 사용할 수 있는 숫자의 개수

n진법 → 0부터 n-1까지 사용 

■ **2진법 (Binary)**

-숫자: 0, 1

-컴퓨터는 모든 데이터를 **2진수**로 처리함

■ **왜 16진법을 쓰는가**

- 2진수는 길어서 보기 불편 → 16진수로 줄여서 표현

## **비트&바이트**

•**비트(bit)**: 0 또는 1 (데이터의 최소 단위)

•**바이트(byte)**: 8비트 = 1바이트

•**MSB**: 가장 왼쪽 비트 → 값에 가장 큰 영향

•**LSB**: 가장 오른쪽 비트 → 값에 가장 작은 영향

**■ 부호 비트**

- MSB로 양수/음수 구분
    
    0 → 양수
    
    1 → 음수
    
- 같은 값도 signed / unsigned에 따라 다르게 해석됨

**■ 바이트 오더링 (엔디안)**

- 데이터 저장 순서 방식

 **• Big Endian:** 큰 값부터 저장

 **• Little Endian:** 작은 값부터 저장

**📌 핵심**

- **메모리 저장 순서는 엔디안에 따라 달라짐**
- **대부분 시스템은 Little Endian 사용**
- **문자열은 순서 그대로 저장됨**
- **정수 데이터는 엔디안 영향 받음**

**ex)**

- 문자열: "ABCD" → 그대로 저장 → 41 42 43 44
- 정수: 0x41424344 → Little Endian(오른쪽부터)→ 44 43 42 41

## 비트 연산

**비트 연산이란?**

데이터를 2진수(비트) 단위로 연산하는 것

**■ 기본 비트 연산자**

- | (OR): 하나라도 1이면 1
- & (AND): 둘 다 1이면 1
- ^ (XOR): 다르면 1
- ~ (NOT): 0↔1 반전

**■ 시프트 연산**

- << n : 왼쪽 이동 → × 2ⁿ
- >> n : 오른쪽 이동 → ÷ 2ⁿ

 **실전에서 중요한 활용**

**📌 1. 비트 마스킹 (AND)**

 특정 비트만 추출

- 0xABCD & 0x00FF = 0xCD
- **필요한 부분만 남기고 나머지 제거**

**📌 2. 특정 바이트 추출**

시프트 + AND 조합

- 하위 1바이트
0x12345678 & 0xFF → 0x78
- 상위 1바이트
0x12345678 >> 24 → 0x12
- **원하는 위치 데이터만 뽑기**

**📌 3. XOR 활용**

값 비교 / 간단한 암호화

- 같은 값 XOR → 0
- x ^ y ^ y = x
- **값 비교 + 복호화 가능**

**📌 4. 시프트로 연산**

 빠른 계산

- x << 1 = x × 2
- x >> 1 = x ÷ 2
- **곱셈/나눗셈 대체 가능**

**■ 인코딩 / 디코딩**

- 인코딩: 데이터를 다른 형식으로 변환
- 디코딩: 다시 원래 값으로 복구
- **인코딩은 누구나 복구 가능 (암호화와 다름)**

**■ 왜 사용하는가**

- 데이터 처리 효율 증가
- 컴퓨터가 이해하기 쉽게 변환

**■ 아스키 코드 (ASCII)**

- 문자를 숫자로 바꾸는 표준
- 1문자 = 1바이트 (0~127)

**EX) A → 65 a → 97**

**■ 유니코드 (Unicode)**

- 전 세계 모든 문자에 번호 부여한 표준
- U+XXXX 형태로 표현
- 모든 언어 표현 가능

**■ UTF-8** 

- 유니코드 인코딩 방식
- 1~4바이트 가변 길이
- 사실상 표준

**■ URL 인코딩**

- URL에서 허용 안 되는 문자 변환
- % + 16진수

**EX)  공백 → %20**

       **, → %2C**

**■ Base64 인코딩**

- 이진 데이터를 텍스트로 변환
- 문자: A-Z, a-z, 0-9, +, /
- 끝에 = 패딩 사용

**EX) dream → ZHJlYW0=**

- 이미지/데이터를 문자열로 표현할 때 사용

## 운영체제

**■ 운영체제란?**

 하드웨어 ↔ 사용자/프로그램 사이 중재, 자원 관리 담당

**■ 역할**

- CPU 할당 (스케줄링)
- 메모리 관리
- 입출력 관리

**■ 구조**

- **커널 (Kernel)** → 하드웨어 직접 관리, 실제 작업 담당
- **셸 (Shell)** → 사용자 ↔ 커널 연결, 명령어 해석

**■ 종류**

- Windows → GUI 기반, 사용 편함
- UNIX / Linux → CUI 기반, Linux는 오픈소스
    - 예: Ubuntu

## **가상 머신**

# **🖥 가상 머신 (VM)**

- 가상 머신(Virtual Machine): 한 컴퓨터 안에서 다른 컴퓨터 환경을 실행하는 소프트웨어
    - 호스트 머신: 실제 컴퓨터
    - 게스트 머신: VM 안에서 실행되는 O
- 필요 이유
    - 여러 운영체제 환경을 한 컴퓨터에서 구축 가능
    - 독립된 환경 제공 → 실습 중 호스트 안전
    - 상태 저장/복원 가능 (스냅샷 기능)
- 주요 VM 환경
    - Mac (Apple Silicon) → UTM으로 Linux 환경
    - Windows → VirtualBox / WSL2로 Linux 환경
    - Apple Silicon에서 x86 GUI 환경은 성능 제한 존재
- 장점
    - 경제적 → 여러 대 컴퓨터 불필요
    - 유연 → OS 버전별 환경 쉽게 구성 가능
    - 안정적 → 호스트 보호 + 스냅샷 활용

## 셸(Shell)

이란? 사용자가 리눅스 시스템과 상호작용할 수 있는 인터페이스

-명령어 입력 → 운영체제가 처리 → 결과 출력

- 실습 환경
    - 우분투 VM에서 셸 사용
    - 터미널 실행 단축키
        - Windows / Linux: Ctrl + Alt + T
        - Mac: Control + Option + T
    - 자동으로 bash 셸 실행
- bash 셸
    - 리눅스/유닉스 시스템에서 가장 많이 쓰이는 셸
    - 기능
        - 명령어 입력 → 시스템 실행 → 결과 출력
        - 스크립트 작성 가능 → 반복 작업 자동화
        - 환경 변수 관리, 파일 탐색, 프로세스 제어 등 지원
    - 실습에서 쓰는 이유
        - 우분투 VM에서 기본으로 제공
        - 터미널 열면 자동 실행됨
        - 대부분의 리눅스 명령어가 Bash 기준으로 동작
    
    핵심: Bash는 리눅스에서 명령어 입력과 스크립트 실행을 담당하는 기본 셸이라고 이해하면 충분
    
- 셸 프롬프트(Shell Prompt)
    - **Ex) user@user-VirtualBox:~$**
    - 명령어 입력 준비 완료 상태 표시
- 명령어 예시
    - id → 현재 유저 ID와 그룹 ID 확인

## **권한 (Permission)**

**1. 유저(user)와 그룹(group)**

- 리눅스는 유저와 그룹 기준으로 권한을 관리함
- 유저 → UID
- 그룹 → GID
- 파일 접근 시 UID와 GID를 확인해서 권한 판단
- 유저 정보: /etc/passwd
- 그룹 정보: /etc/group

**2. 권한 종류**

- r → 읽기
- w → 쓰기 (수정/삭제)
- x → 실행 (디렉토리는 접근)

**3. 권한 확인**

명령어: ls -l

Ex)  -rwxrw-r–
구조[파일타입][소유자 권한][그룹 권한][기타 권한]

- [-] : 일반 파일
- d : 디렉토리
- l : 링크
- rwx → 소유자 권한
- rw- → 그룹 권한
- r– → 기타 사용자 권한

**4. 숫자 권한**

- r = 4
- w = 2
- x = 1

조합

- rwx = 7
- rw- = 6
- r– = 4

Ex)

- 764 → rwx rw- r–

**5. chmod (권한 변경)**

- 숫자로 변경
chmod 764 파일명

- 문자로 변경

chmod g+x 파일명   → 그룹에 실행 추가
chmod g-wx 파일명  → 그룹에서 쓰기/실행 제거

- u → 사용자
- g → 그룹
- o → 기타

**6. chown (소유자 변경)**

- sudo chown 사용자 파일명
- root 권한 필요
- 소유자가 바뀌면 접근 권한도 달라짐

**7. 특수 권한**

- setuid (4)
실행 시 파일 소유자 권한으로 실행
- setgid (2)
실행 시 그룹 권한으로 실행
- sticky bit (1)
디렉토리에서 본인 파일만 삭제 가능

# 4.드림핵의 주요 학습 카테고리

## **1. Web Hacking**

웹 해킹이란?

-웹 상에서 본래의 의도와 다른 동작을 일으키거나 데이터를 도용, 변조, 시스템을 손상시키는 등의 악의적인 행위를 수행하는 것

### **웹 해킹 공부 전 필요한 기초 지식**

#### 1) 웹 브라우저–서버 프로토콜 약자

- **HTTP (HyperText Transfer Protocol)**
 웹에서 요청/응답 주고받는 규칙
- 브라우저 → 요청(Request)
- 서버 → 응답(Response)

#### 2) HTTPS의 ‘S’ 의미

- S= secure
- HTTPS=HTTP+TLS(암호화)

#### 3) 로그인 상태 유지 방법

- 쿠키+세션 구조

서버:세션 저장

클라이언트: 쿠키에 세션 ID 저장

- 흐름

로그인 성공-세션 생성-쿠키로 session ID 저장-이후 요청마다 자동 포함

#### 4)요청 성공 시 상태 코드

- 200 OK
- 201 created

#### 5) 클라이언트 오류 상태 코드

- 400 Bad Request
- 401 Unauthorized
- 403 Forbidden(권한 없이 접근)
- 404 Not Found(URL 오류)

#### 10) GET vs POST 차이

GET

- 조회용

POST

서버에 **데이터를 보내서 생성/처리(Create)** 하는 요청

- Body에 데이터 포함
- 데이터 전송용
- 보안 상대적으로 높음
- 요청할 때마다 결과 달라질 수 있음

## 4. Cryptography(암호학)

**암호학이란?**

- 정보를 보호하거나 안전하게 통신하기 위한 학문
- 키(key)를 이용하며 평문(Plaintext)을 암호문(Ciphertext)으로 변환하거나, 암호문을 평문으로 변환하는 전반적인 과정
- 개인정보 관리, 전자상거래, 클라우드 서비스 등 보안을 유지해야 하는 다양한 영역에서 사용
  
**추가 개념**

 Plaintext(평문): 암호화되기 전의 원래 데이터 즉, 원본 메시지
 Ciphertext(암호문): 암호화된 결과 데이터, 키 없으면 해석 불가
Key: 암호화/복호화에 사용하는 비밀 값, Plaintext를 Ciphertext 로 바꾸는 기준 또는 반대로 복구하는 열쇠

### **암호학 공부 전 필요한 기초 지식**

#### (1) 비트 연산 정의와 다양한 비트 연산자가 각 어떤 연산을 하는가

비트 연산: 0과1(비트단위)로 하는 연산

주요 연산자

- &(AND): 둘 다 1일 때만 1
- |(OR): 하나라도 1이면 1
- ^(XOR): 다르면 1, 같으면 0
- ~(NOT): 0↔1 반전
- <<: 왼쪽으로 이동(값 증가)
- >>: 오른쪽으로 이동(값 감소)

#### (2-1) 모듈로 연산

나눴을 때 나머지를 구하는 연산

(a mod m=a를 m으로 나눈 나머지)

#### 왜 필요한가?

👉 암호학에서는 숫자가 너무 커지니까

- 계속 줄여줘야 계산 가능함
- 그래서 “중간중간 나머지로 줄임”

👉 **mod는 “계산 결과를 일정 범위 안으로 계속 줄이는 장치”**

#### (2-2) 모듈로 연산에서의 덧셈/뺄셈/곱셈

일반 계산 후 마지막에 mod 적용

- (a + b) mod m
- (a - b) mod m
- (a x b) mod m

#### (2-3) 곱셈 역원 & 나눗셈

- 곱셈 역원

어떤 수 a에 대해 (a × x) mod m=1이 되는 x

- 나눗셈

곱셈 역원으로 변환해서 계산(그냥 나누기X)

a / b mod m

→ a x (b의 역원) mod m

“모듈로에서는 나눗셈=역원 곱하기”

#### (3) 거듭제곱 & 빠른 계산

기본 문제: a^b mod m →*b가 크면 계산 불가능

해결(빠른 거듭제곱)

“지수를 반으로 줄이면서 계산”

- b가 짝수 —> (a^(b/2))²
- b가 홀수 —> a x a^(b-1)

### **1) SSH 사용법**

#### (1) 기본 접속 명령어
`ssh user@host -p port`

#### (2) 구조

- `user` → 접속 계정 이름
- `host` → 서버 주소 (도메인 or IP)
- `-p port` → 접속 포트 지정
- `-i` → 인증에 사용할 **개인키 파일 지정**

#### (3) 접속 과정

명령어 입력→보안 확인 메시지 뜸 (yes 입력)→비밀번호 입력 (또는 키 인증)→리눅스 쉘 접속 완료

#### (4) 인증 방식

**패스워드 인증**

👉 접속 후 비밀번호 입력으로 인증

- `ssh user@HOST` 실행
- 서버가 비밀번호 요구
- 맞으면 로그인 성공

**공개 키 인증**

👉 키 쌍으로 인증

- 클라이언트: 개인키
- 서버: 공개키

👉 공개 키-개인 키 키쌍은 ssh-keygen 명령으로 생성 가능

#### (4) Docker(도커)
컨테이너를 만들고, 실행하고, 배포할 수 있는 가상화 플랫폼

도커의 컨테이너란?

-가상의 환경이 구축되어 있는 하나의 박스

-새로운 운영체제 환경 구축 필요 X, 하나의 분리된 프로세스처럼 작동 즉, 특정한 환경을 구성하기 위해 만들어진 가상의 공간

- **도커 이미지(Docker Image)**

-실행 설계도(ex:컨테이너 생성에 필요한 파일, 환경 변수, 명령어 등과 파일 시스템 포함)

-이미지 생성 위해선 Dockerfile(이미지를 빌드하는 데 단계적으로 필요한 명령어가 있는 파일)을 작성하고 이미지를 빌드해야 함

-도커 이미지에는 태그(Tag) 붙이기 가능 주로 이미지의 버전을 지정하기 위해 사용

- 도커 컨테이너(Docker Container)

-도커 이미지가 실행된 상태로 만들어진 독립된 실행 환경

-이미지 실행하면 컨테이너 생성

-실행할 때 지정한 옵션에 따라 동작, 각 컨테이너는 서로 분리된 파일 시스템을 사용해 독립적으로 동작

- 도커 레지스트리(Docker Registry)

-도커 이미지 저장하는 저장소(도커의 공식 레지스트리: Docker hub)

-누구나 도커 이미지를 올리고, 존재하는 도커 이미지 가져오기 가능

**도커를 쓰는 이유는?**

-가상 환경 구축의 많은 부분을 생략하고 가볍게 실행됨 → 간편하게 문제 환경 재현, 문제 풀이에 방해가 되는 요소 줄임

-문제가 각각 분리된 환경에 있는 것처럼 제공하여 다른 문제에 영향을 미치는 행위 혹은 출제자가 의도하지 않은 공격 수행 등을 막음

#### 1) 도커 명령어

- **도커 명령어 - 1**

**docker build 기본 명령어**

```c
docker build [옵션] <경로>
docker build -t <이미지명:태그> <경로>
```

1. -t 옵션으로 이미지의 이름과 태그 지정, 작성하지 않을 경우 ‘latest’로 지정
2. docker build . : 현재 디렉토리에 있는 Dockerfile로 이미지 생성
3. docker build -t my-image: 현재 디렉토리에 있는 Dockerfile로 ‘my-image:latest’ 이미지 생성
- **docker images**

도커 이미지 목록 출력 명령어

- **도커 명령어 - 2**

**docker run**

도커 이미지로 컨테이너 생성 및 실행

```c
docker run [옵션] <이미지명|ID> <명령어>
docker run -p <호스트 PORT>:<컨테이너 PORT> <이미지명|ID>
docker run -it <이미지명|ID> <명령어>
```

**-p 옵션**: 도커 컨테이너의 포트와 호스트의 포트를 매핑

(포트 매핑: 컨테이너 안의 서비스 포트를, 내 컴퓨터(호스트) 포트랑 연결하는 것

📌 왜 필요하냐?

컨테이너 안에서는 웹 서버가 이렇게 떠있음:

- 컨테이너 내부: 80번 포트에서 서버 실행

근데 문제 👉 내 PC에서는 컨테이너 내부 포트에 직접 접근 못함 그래서 연결해 줘야 함)

**-it 옵션:** 컨테이너에서 bash 셸 사용 가능

-i (--interactive): 표준 입력 활성화하여 사용자가 명령어 입력 가능하게 함

-t (--tty): 가상 터미널(tty) 사용 가능하게 함

ex) docker run -it my-image:1 /bin/bash :

my image:1 이미지로 컨테이너를 생성하고 실행하여 bash 셸 열기

 **docker ps**

실행 중인 컨테이너 목록 출력

```c
docker ps -a
```

-**a 옵션:** 종료된 컨테이너까지 모두 출력

- **도커 명령어 - 3**

docker run 대신 컨테이너 생성과 실행을 따로 명령할 수 있다

**docker create**

도커 이미지로 컨테이너 생성

**docker start**

중단된 컨테이너 시작

```c
docker create [옵션] <이미지명|ID> [명령어]
docker start [옵션] <컨테이너명|ID]
```

**docker exec**

실행 중인 컨테이너에 접속하여 명령 수행

```c
docker exec [옵션] <컨테이너명|ID> [명령어]
```

docker run과 유사하게 -it 옵션으로 bash 셸 실행 가능

→ docker exec -it <컨테이너명|ID> /bin/bash

: 실행 중인 컨테이너에서 bash 셸 열기

**docker stop**

실행 중인 컨테이너 중단

```c
docker stop [옵션] <컨테이너명|ID>
```

- **도커 명령어 - 4**

**docker pull**

레지스트리에 존재하는 도커 이미지 다운받음

**docker rm**

도커 컨테이너 삭제

**docker rmi**

도커 이미지 삭제

**docker inspect**

도커 이미지 or 컨테이너의 자세한 정보 출력

```c
docker pull [옵션] <이미지명>
docker rm [옵션] <컨테이너명|ID>
docker rmi [옵션] <이미지명|ID>
docker inspect [옵션] <이미지 or 컨테이너명|ID>
```

(2) Dockerfile 명령어

-Dockerfile은 FROM 명령어로 시작 이후 순서대로 명령어 실행

- **FROM**

-생성할 이미지의 기반이 되는 base 이미지 지정

-보통 사용할 운영체제의 공식 이미지를 Dockerhub 에서 가져옴

```c
FROM 이미지:태그
```

`FROM` = 기반 이미지 지정

`ubuntu` = 이미지 이름

`18.04` = 태그(버전)

- **ENV**

```c
ENV 변수명 값
ENV 변수명=값

```

-Dockerfile 내에서 사용하는 환경 변수 지정

-파일 내에서 변수는 $변수명 or ${변수명} 형태로 표현

EX)

ENV PYTHON_VERSION 3.11.2

→ …/python/$PYTHON_VERSION/…

**(2) Dockerfile 명령어 -2**

- **RUN**

-이미지를 빌드할 때 실행할 명령어 작성

-필요한 패키지 설치 or 파일 권한 설정 등의 작업 수행

```c
RUN 명령어 -> RUN apt-get update
RUN ["명령어", "인자1", "인자2"]
-> RUN ["/bin/bash", "-c", "echo hello"]
```

- **COPY**

-src 파일이나 디렉토리를 이미지 파일 시스템의 dest로 복사

```c
COPY <src> <dest>
```

EX)

copy . /app

의미: 현재 폴더(`.`) 안의 모든 파일을 Docker 이미지 내부의 `/app` 폴더로 복사

- ADD

-src 파일이나 디렉토리, URL을 이미지 파일 시스템의 dest로 복사

```c
ADD <src> <dest>
ex) ADD . /app
```

- WORKDIR

-Dockerfile 내의 명령을 수행할 작업 디렉토리 지정

-리눅스의 cd 명령어와 유사

```c
WORKDIR 디렉토리
ex) WORKDIR /home/user
```

- USER

-명령을 수행할 사용자 or 그룹 지정

```c
USER 사용자명 |UID
USER 사용자명 |UID[:그룹명|GID]
ex) USER $username
```

- EXPOSE

-컨테이너가 실행 중일 때 들어오는 네트워크 트래픽을 리슨할 포트와 프로토콜 지정

-사용할 수 있는 프로토콜: TCP, UDP 기본적으론 TCP 지정됨

```c
EXPOSE 포트
EXPOSE 포트/프로토콜
ex) EXPOSE 80/tcp
```

**(3) Dockerfile 명령어 -3**

- ENTRYPOINT

-컨테이너가 실행될 때 수행할 명령어 지정

```c
ENTRYPOINT 명령어 인자1 인자2
ENTRYPOINT ["명령어", "인자1", "인자2"]
ex) ENTRYPOINT ["echo", "hello"]
```

- CMD

-컨테이너가 실행될 때 수행할 명령어 지정 or ENTERPOINT 명령어에 인자 전달

```c
CMD 명령어 인자1 인자2
CMD ["명령어", "인자1", "인자2"]
CMD ["인자1", "인자2"]
ex) CMD ["echo", "hello"]
```

-Dockerfile 내에 CMD 명령이 여러 개 존재하면 마지막 CMD 사용

-docker run의 인자를 작성하면 CMD 명령어 무시

-ENTERPOINT가 있는 경우, docker run의 인자가 ENTERPOINT의 인자로 들어감

```c
ENTRYPOINT ["python"]
CMD ["[app.py](http://app.py/)"]
```

Dockerfile이 위와 같을 때

'''docker run [이미지]로 컨테이너 실행 시 python [app.py](http://app.py) 실행'''

'''docker run [이미지] [test.py](http://test.py)로 컨테이너 실행 시 python test.py 실행'''
