# **1.컴퓨터 과학 기초**

## **진법이란?**

: 한 자릿수에 사용할 수 있는 숫자의 개수

n진법 → 0부터 n-1까지 사용 

■ **2진법 (Binary)**

-숫자: 0, 1

-컴퓨터는 모든 데이터를 **2진수**로 처리함

■ **왜 16진법을 쓰는가**

- 2진수는 길어서 보기 불편 → 16진수로 줄여서 표현

## **비트&바이트**

•**비트(bit)**: 0 또는 1 (데이터의 최소 단위)

•**바이트(byte)**: 8비트 = 1바이트

•**MSB**: 가장 왼쪽 비트 → 값에 가장 큰 영향

•**LSB**: 가장 오른쪽 비트 → 값에 가장 작은 영향

**■ 부호 비트**

- MSB로 양수/음수 구분
    
    0 → 양수
    
    1 → 음수
    
- 같은 값도 signed / unsigned에 따라 다르게 해석됨

**■ 바이트 오더링 (엔디안)**

- 데이터 저장 순서 방식

 **• Big Endian:** 큰 값부터 저장

 **• Little Endian:** 작은 값부터 저장

**📌 핵심**

- **메모리 저장 순서는 엔디안에 따라 달라짐**
- **대부분 시스템은 Little Endian 사용**
- **문자열은 순서 그대로 저장됨**
- **정수 데이터는 엔디안 영향 받음**

**ex)**

- 문자열: "ABCD" → 그대로 저장 → 41 42 43 44
- 정수: 0x41424344 → Little Endian(오른쪽부터)→ 44 43 42 41

## 비트 연산

**비트 연산이란?**

데이터를 2진수(비트) 단위로 연산하는 것

**■ 기본 비트 연산자**

- | (OR): 하나라도 1이면 1
- & (AND): 둘 다 1이면 1
- ^ (XOR): 다르면 1
- ~ (NOT): 0↔1 반전

**■ 시프트 연산**

- << n : 왼쪽 이동 → × 2ⁿ
- >> n : 오른쪽 이동 → ÷ 2ⁿ

 **실전에서 중요한 활용**

**📌 1. 비트 마스킹 (AND)**

 특정 비트만 추출

- 0xABCD & 0x00FF = 0xCD
- **필요한 부분만 남기고 나머지 제거**

**📌 2. 특정 바이트 추출**

시프트 + AND 조합

- 하위 1바이트
0x12345678 & 0xFF → 0x78
- 상위 1바이트
0x12345678 >> 24 → 0x12
- **원하는 위치 데이터만 뽑기**

**📌 3. XOR 활용**

값 비교 / 간단한 암호화

- 같은 값 XOR → 0
- x ^ y ^ y = x
- **값 비교 + 복호화 가능**

**📌 4. 시프트로 연산**

 빠른 계산

- x << 1 = x × 2
- x >> 1 = x ÷ 2
- **곱셈/나눗셈 대체 가능**

**■ 인코딩 / 디코딩**

- 인코딩: 데이터를 다른 형식으로 변환
- 디코딩: 다시 원래 값으로 복구
- **인코딩은 누구나 복구 가능 (암호화와 다름)**

**■ 왜 사용하는가**

- 데이터 처리 효율 증가
- 컴퓨터가 이해하기 쉽게 변환

**■ 아스키 코드 (ASCII)**

- 문자를 숫자로 바꾸는 표준
- 1문자 = 1바이트 (0~127)

**EX) A → 65 a → 97**

**■ 유니코드 (Unicode)**

- 전 세계 모든 문자에 번호 부여한 표준
- U+XXXX 형태로 표현
- 모든 언어 표현 가능

**■ UTF-8** 

- 유니코드 인코딩 방식
- 1~4바이트 가변 길이
- 사실상 표준

**■ URL 인코딩**

- URL에서 허용 안 되는 문자 변환
- % + 16진수

**EX)  공백 → %20**

       **, → %2C**

**■ Base64 인코딩**

- 이진 데이터를 텍스트로 변환
- 문자: A-Z, a-z, 0-9, +, /
- 끝에 = 패딩 사용

**EX) dream → ZHJlYW0=**

- 이미지/데이터를 문자열로 표현할 때 사용

## 운영체제

**■ 운영체제란?**

 하드웨어 ↔ 사용자/프로그램 사이 중재, 자원 관리 담당

**■ 역할**

- CPU 할당 (스케줄링)
- 메모리 관리
- 입출력 관리

**■ 구조**

- **커널 (Kernel)** → 하드웨어 직접 관리, 실제 작업 담당
- **셸 (Shell)** → 사용자 ↔ 커널 연결, 명령어 해석

**■ 종류**

- Windows → GUI 기반, 사용 편함
- UNIX / Linux → CUI 기반, Linux는 오픈소스
    - 예: Ubuntu

## **가상 머신**

# **🖥 가상 머신 (VM)**

- 가상 머신(Virtual Machine): 한 컴퓨터 안에서 다른 컴퓨터 환경을 실행하는 소프트웨어
    - 호스트 머신: 실제 컴퓨터
    - 게스트 머신: VM 안에서 실행되는 O
- 필요 이유
    - 여러 운영체제 환경을 한 컴퓨터에서 구축 가능
    - 독립된 환경 제공 → 실습 중 호스트 안전
    - 상태 저장/복원 가능 (스냅샷 기능)
- 주요 VM 환경
    - Mac (Apple Silicon) → UTM으로 Linux 환경
    - Windows → VirtualBox / WSL2로 Linux 환경
    - Apple Silicon에서 x86 GUI 환경은 성능 제한 존재
- 장점
    - 경제적 → 여러 대 컴퓨터 불필요
    - 유연 → OS 버전별 환경 쉽게 구성 가능
    - 안정적 → 호스트 보호 + 스냅샷 활용

## 셸(Shell)

이란? 사용자가 리눅스 시스템과 상호작용할 수 있는 인터페이스

-명령어 입력 → 운영체제가 처리 → 결과 출력

- 실습 환경
    - 우분투 VM에서 셸 사용
    - 터미널 실행 단축키
        - Windows / Linux: Ctrl + Alt + T
        - Mac: Control + Option + T
    - 자동으로 bash 셸 실행
- bash 셸
    - 리눅스/유닉스 시스템에서 가장 많이 쓰이는 셸
    - 기능
        - 명령어 입력 → 시스템 실행 → 결과 출력
        - 스크립트 작성 가능 → 반복 작업 자동화
        - 환경 변수 관리, 파일 탐색, 프로세스 제어 등 지원
    - 실습에서 쓰는 이유
        - 우분투 VM에서 기본으로 제공
        - 터미널 열면 자동 실행됨
        - 대부분의 리눅스 명령어가 Bash 기준으로 동작
    
    핵심: Bash는 리눅스에서 명령어 입력과 스크립트 실행을 담당하는 기본 셸이라고 이해하면 충분
    
- 셸 프롬프트(Shell Prompt)
    - **Ex) user@user-VirtualBox:~$**
    - 명령어 입력 준비 완료 상태 표시
- 명령어 예시
    - id → 현재 유저 ID와 그룹 ID 확인

## **권한 (Permission)**

**1. 유저(user)와 그룹(group)**

- 리눅스는 유저와 그룹 기준으로 권한을 관리함
- 유저 → UID
- 그룹 → GID
- 파일 접근 시 UID와 GID를 확인해서 권한 판단
- 유저 정보: /etc/passwd
- 그룹 정보: /etc/group

**2. 권한 종류**

- r → 읽기
- w → 쓰기 (수정/삭제)
- x → 실행 (디렉토리는 접근)

**3. 권한 확인**

명령어: ls -l

Ex)  -rwxrw-r–
구조[파일타입][소유자 권한][그룹 권한][기타 권한]

- [-] : 일반 파일
- d : 디렉토리
- l : 링크
- rwx → 소유자 권한
- rw- → 그룹 권한
- r– → 기타 사용자 권한

**4. 숫자 권한**

- r = 4
- w = 2
- x = 1

조합

- rwx = 7
- rw- = 6
- r– = 4

Ex)

- 764 → rwx rw- r–

**5. chmod (권한 변경)**

- 숫자로 변경
chmod 764 파일명

- 문자로 변경

chmod g+x 파일명   → 그룹에 실행 추가
chmod g-wx 파일명  → 그룹에서 쓰기/실행 제거

- u → 사용자
- g → 그룹
- o → 기타

**6. chown (소유자 변경)**

- sudo chown 사용자 파일명
- root 권한 필요
- 소유자가 바뀌면 접근 권한도 달라짐

**7. 특수 권한**

- setuid (4)
실행 시 파일 소유자 권한으로 실행
- setgid (2)
실행 시 그룹 권한으로 실행
- sticky bit (1)
디렉토리에서 본인 파일만 삭제 가능

# 4.드림핵의 주요 학습 카테고리

## **1. Web Hacking**

웹 해킹이란?

-웹 상에서 본래의 의도와 다른 동작을 일으키거나 데이터를 도용, 변조, 시스템을 손상시키는 등의 악의적인 행위를 수행하는 것

### **웹 해킹 공부 전 필요한 기초 지식**

#### 1) 웹 브라우저–서버 프로토콜 약자

- **HTTP (HyperText Transfer Protocol)**
 웹에서 요청/응답 주고받는 규칙
- 브라우저 → 요청(Request)
- 서버 → 응답(Response)

#### 2) HTTPS의 ‘S’ 의미

- S= secure
- HTTPS=HTTP+TLS(암호화)

#### 3) 로그인 상태 유지 방법

- 쿠키+세션 구조

서버:세션 저장

클라이언트: 쿠키에 세션 ID 저장

- 흐름

로그인 성공-세션 생성-쿠키로 session ID 저장-이후 요청마다 자동 포함

#### 4)요청 성공 시 상태 코드

- 200 OK
- 201 created

#### 5) 클라이언트 오류 상태 코드

- 400 Bad Request
- 401 Unauthorized
- 403 Forbidden(권한 없이 접근)
- 404 Not Found(URL 오류)

POST

서버에 **데이터를 보내서 생성/처리(Create)** 하는 요청

- Body에 데이터 포함
- 데이터 전송용

## 4. Cryptography(암호학)

**암호학이란?**

- 정보를 보호하거나 안전하게 통신하기 위한 학문
- 키(key)를 이용하며 평문(Plaintext)을 암호문(Ciphertext)으로 변환하거나, 암호문을 평문으로 변환하는 전반적인 과정
- 개인정보 관리, 전자상거래, 클라우드 서비스 등 보안을 유지해야 하는 다양한 영역에서 사용
  
**추가 개념**

 Plaintext(평문): 암호화되기 전의 원래 데이터 즉, 원본 메시지
 Ciphertext(암호문): 암호화된 결과 데이터, 키 없으면 해석 불가
Key: 암호화/복호화에 사용하는 비밀 값, Plaintext를 Ciphertext 로 바꾸는 기준 또는 반대로 복구하는 열쇠

### **암호학 공부 전 필요한 기초 지식**

#### (1) 비트 연산 정의와 다양한 비트 연산자가 각 어떤 연산을 하는가

비트 연산: 0과1(비트단위)로 하는 연산

주요 연산자

- &(AND): 둘 다 1일 때만 1
- |(OR): 하나라도 1이면 1
- ^(XOR): 다르면 1, 같으면 0
- ~(NOT): 0↔1 반전
- <<: 왼쪽으로 이동(값 증가)
- >>: 오른쪽으로 이동(값 감소)

#### (2-1) 모듈로 연산

나눴을 때 나머지를 구하는 연산

(a mod m=a를 m으로 나눈 나머지)

#### 왜 필요한가?

👉 암호학에서는 숫자가 너무 커지니까

- 계속 줄여줘야 계산 가능함
- 그래서 “중간중간 나머지로 줄임”

👉 **mod는 “계산 결과를 일정 범위 안으로 계속 줄이는 장치”**

#### (2-2) 모듈로 연산에서의 덧셈/뺄셈/곱셈

일반 계산 후 마지막에 mod 적용

- (a + b) mod m
- (a - b) mod m
- (a x b) mod m

#### (2-3) 곱셈 역원 & 나눗셈

- 곱셈 역원

어떤 수 a에 대해 (a × x) mod m=1이 되는 x

- 나눗셈

곱셈 역원으로 변환해서 계산(그냥 나누기X)

a / b mod m

→ a x (b의 역원) mod m

“모듈로에서는 나눗셈=역원 곱하기”

#### (3) 거듭제곱 & 빠른 계산

기본 문제: a^b mod m →*b가 크면 계산 불가능

해결(빠른 거듭제곱)

“지수를 반으로 줄이면서 계산”

- b가 짝수 —> (a^(b/2))²
- b가 홀수 —> a x a^(b-1)

### **1) SSH 사용법**

#### (1) 기본 접속 명령어
`ssh user@host -p port`

#### (2) 구조

- `user` → 접속 계정 이름
- `host` → 서버 주소 (도메인 or IP)
- `-p port` → 접속 포트 지정
- `-i` → 인증에 사용할 **개인키 파일 지정**

#### (3) 접속 과정

명령어 입력→보안 확인 메시지 뜸 (yes 입력)→비밀번호 입력 (또는 키 인증)→리눅스 쉘 접속 완료

#### (4) 인증 방식

**패스워드 인증**

👉 접속 후 비밀번호 입력으로 인증

- `ssh user@HOST` 실행
- 서버가 비밀번호 요구
- 맞으면 로그인 성공

**공개 키 인증**

👉 키 쌍으로 인증

- 클라이언트: 개인키
- 서버: 공개키

👉 공개 키-개인 키 키쌍은 ssh-keygen 명령으로 생성 가능

#### (4) Docker(도커)
컨테이너를 만들고, 실행하고, 배포할 수 있는 가상화 플랫폼

도커의 컨테이너란?

-가상의 환경이 구축되어 있는 하나의 박스

-새로운 운영체제 환경 구축 필요 X, 하나의 분리된 프로세스처럼 작동 즉, 특정한 환경을 구성하기 위해 만들어진 가상의 공간

- **도커 이미지(Docker Image)**

-실행 설계도(ex:컨테이너 생성에 필요한 파일, 환경 변수, 명령어 등과 파일 시스템 포함)

-이미지 생성 위해선 Dockerfile(이미지를 빌드하는 데 단계적으로 필요한 명령어가 있는 파일)을 작성하고 이미지를 빌드해야 함

-도커 이미지에는 태그(Tag) 붙이기 가능 주로 이미지의 버전을 지정하기 위해 사용

- 도커 컨테이너(Docker Container)

-도커 이미지가 실행된 상태로 만들어진 독립된 실행 환경

-이미지 실행하면 컨테이너 생성

-실행할 때 지정한 옵션에 따라 동작, 각 컨테이너는 서로 분리된 파일 시스템을 사용해 독립적으로 동작

- 도커 레지스트리(Docker Registry)

-도커 이미지 저장하는 저장소(도커의 공식 레지스트리: Docker hub)

-누구나 도커 이미지를 올리고, 존재하는 도커 이미지 가져오기 가능

**도커를 쓰는 이유는?**

-가상 환경 구축의 많은 부분을 생략하고 가볍게 실행됨 → 간편하게 문제 환경 재현, 문제 풀이에 방해가 되는 요소 줄임

-문제가 각각 분리된 환경에 있는 것처럼 제공하여 다른 문제에 영향을 미치는 행위 혹은 출제자가 의도하지 않은 공격 수행 등을 막음

#### 1) 도커 명령어

**docker build 기본 명령어**

```

1. -t 옵션으로 이미지의 이름과 태그 지정, 작성하지 않을 경우 ‘latest’로 지정
2. docker build . : 현재 디렉토리에 있는 Dockerfile로 이미지 생성
3. docker build -t my-image: 현재 디렉토리에 있는 Dockerfile로 ‘my-image:latest’ 이미지 생성

-t (--tty): 가상 터미널(tty) 사용 가능하게 함

my image:1 이미지로 컨테이너를 생성하고 실행하여 bash 셸 열기

#### (5) 정규 표현식

**정규 표현식이란?**

특정한 패턴으로 문자열을 표현하는 식. 일반적으로 문자열 내에서 원하는 패턴의 문자열을 찾거나 치환할 때 사용.

-간단하게 특정 형식의 문자열을 표현하고, 형식이 일치하는지 검증할 수 있음

-입력 값을 검증하는 데 활용되며, 잘못된 정규 표현식으로 잘못 검증 시 취약점으로 이어짐

-취약점을 찾고 이를 보안하는 해킹 분야에서 반드시 알아야 할 대상

이메일 정규 표현식 예시

```c
^[\w-\.]+@([\w-]+\.)+[\w-]{2,4}$
```

xxx@xxx.xxx ← 정규 표현식에 매치

xxx@xxx.x ← 정규 표현식에 매치하지 않음

**정규 표현식의 필요성**

-주로 로직 수행 전 이용자가 입력한 값을 검증하기 위해 사용

 -이메일, 전화번호, 주민등록번호, 웹사이트 주소 등 입력 값이 특정 형식이어야 하는 모든 경우에 유용하게 쓰임

-매우 긴 문자열에서 원하는 형식의 문자열을 검색하여 추출할 때도 사용

→정규 표현식 DH\{.+\} 를 해당 출력 값에 매치 시 플래그 쉽게 찾기 가능

#### 1) 정규 표현식 구성

정규 표현식 패턴

정규 표현식은 ‘패턴’ or /패턴/ 의 형태로 작성

/로 패턴을 감싸는 경우 /뒤에 플래그 작성 가능

| 패턴 | 설명 | 예시 |
| --- | --- | --- |
| 문자 or 문자열 | 해당 문자 or 문자열과 매치 | dreamhack → Hello ***dreamhack***
/c/ → Hello dreamhack |
| . | 모든 문자와 매치 | d.eamhack → Hello ***dxeamhack*** |
| \| | 앞 or 뒤의 패턴과 매치 | hi|dream(hi  or dream) →Hello ***dream***hack |
| [] | [] 안 문자와 매치 | [aeiou]→H***i*** |
| [^ ] | ^ 뒤의 패턴을 제외한 나머지와 매치 | [^aeoiu]→***H***i |
| ^ | 어떤 문자열의 시작이 특정 패턴인 경우 매치 | ^dream→***dream***hack Helllo /^dream→ Hello dreamhack(X) |

**re 모듈 대표 함수**

| 함수 | 설명 | 반환 값 |
| --- | --- | --- |
| re.compile(pattern, flags) | 정규 표현식 패턴을 컴파일 | 패턴 객체 반환. 만들어진 패턴 객체는 변수에 저장하여 사용 가능 |
| re.search(pattern, strings, falgs) | 문자열 내에서 패턴에 처음으로 매치하는 문자열 검색 | (매치하는 문자열 O → 문자열 객체 반환, 매치하는 문자열 X → None 반환, 문자열 객체의 group() 매소드를 사용하여 객체에서 문자열만 반환 가능 |
| re.match(pattern, strings, flags) | 문자열 시작 부분에서 패턴에 매치하는 문자열 검색 | ^ 위와 동일 |

| re.finditer(pattern, strings, flags) | ^ 위와 동일 | 모든 문자열 객체를 리스트 형식으로 반환 |

**Raw String (r-string)**

- 이스케이프 문자는 백슬래쉬(\) 뒤에 문자를 붙여 나타냄 (ex: \t, \n) But, 정규 표현식 패턴 내에 백슬래쉬 사용 시 이스케이프 문자로 인식됨 → 패턴 의도와 다르게 해석 → 이를 막기 위해 패턴 작성 시 Raw String 이용
- 문자열 앞에 ‘r’을 붙여 나타냄
- 이스케이프 문자를 문자 그대로 인식하는 특수한 문자열

Raw String 예시

```c
1 print('\t'+'Hi')
2 # 결과:        Hi
3
4 # Raw String
5 print(r'\t'+'Hi')
6 # 결과: \tHi
```

| .exe | 실행 파일 | Windows 운영체제에서 실행 가능한 프로그램 파일. 응용 프로그램 or 설치 프로그램 실행에 사용 |
| .msi | 실행 파일 | Windows Installer 패키지 파일. 소프트웨어 설치 및 배포에 사용. |
| .bat | 실행 파일 | 여러 명령어를 순차적으로 실행할 수 있는 배치 파일. Windows 환경에서 자동화 작업 등에 사용됨. |
| .zip | 압축 파일 | 여러 파일이나 폴더를 하나의 파일로 묶어 저장. 용량을 줄이거나 전송 시 편리하게 사용됨. |
- 확장자는 일종의 약속이며 파일의 이름일 뿐, 그림 파일이나 PDF 파일의 확장자를 .exe로 변경한다고 해서 이 파일이 실제로 실행할 수 있는 파일이 되는 건 아님.
- 확장자는 사람이 인지하기 쉽도록 존재하는 보조 도구이며 운영체제가 이 파일을 어떻게 처리하면 될지 간단하게 알려 주는 용도.

**확장자의 이름에 따라 파일 데이터가 변하지는 않지만, Windows에서는 몇 가지 확장자에 대해 기본적으로 연결할 프로그램을 지정해 주는 방식을 사용함**

예를 들어 Windows는

파일명이 hello.txt → 메모장에 연결하여 열도록 함

같은 파일의 이름을 hello.html로 변경 → 웹 브라우저를 통해 열도록 함

왜?

.txt나 .html 확장자를 가진 파일들은 단순히 데이터가 저장된 파일. 파일 독립적으로 무언가를 할 수 없고, 메모장이나 웹 브라우저와 같은 다른 프로그램을 통해 해석하는 과정 필요. (.txt나 .html 뿐 아니라 대부분의 확장자에서 그러함)

**그럼 exe는?**

.exe → ‘실행 파일’ 가리키는 확장자

-.exe 파일에는 컴퓨터가 이해할 수 있는 명령어들이 작성되어 있음

### (3) 데이터 저장 방식과 16진수 표현

-실제 컴퓨터: 2진수로 데이터 저장 But, 사람이 보다 이해하기 쉽게 16진수로 나타내는 경우 많음(디버깅 도구가 메모리의 위치를 나타내는 주소값을 출력할 때, RGB 색상을 표현할 때 등)

1    2진수 : 1011 1110 1001₂
2                B        E       9
3   16진수 :  0xBE9

- 16진수 표현 시 0x 라는 접두어 붙임

예) 12(10진수) → 0xC(16진수)

**-ASCII(아스키)**

- 컴퓨터가 문자를 숫자로 저장하고 처리할 수 있도록 표준화 된 문자 인코딩 방식
- 알파벳 대소문자, 숫자, 기본적인 특수문자, 여러 제어문자(줄바꿈,탭 등) 포함됨

-**.txt 파일**에도 동일하게 2진수로 데이터가 저장되어 있으나, 메모장은 이를 ASCII 같은 인코딩 체계 중 하나인 유니코드로 보여 줌

### **(4) HxD**

**HxD 편집기란?**

파일에 담긴 데이터를 ASCII 등으로 인코딩 X → 날것의 데이터 그대로를 확인하고, 고칠 수 있는 편집기

### Header(헤더)

Header란?

-파일이나 데이터 구조의 맨 앞 부분에 위치하는 메타데이터. 해당 파일의 형식, 구조, 속성 등에 대한 정보가 담김

-Windows 실행 파일(EXE, DLL 등)의 경우, ‘MZ 헤더’ 라고 불리는 헤더가 있는데 이는 4D 5A(ASCLL로 ‘MZ’)로 시작

-’MZ 식별자’: 이 파일이 실행 파일임을 의미

-헤더에는 식별자 뿐 아니라 프로그램이 메모리에 어떻게 로드되어야 하는지, 어떤 리소스를 사용하는지, 어디서 실행을 시작해야 하는지 등 중요한 정보가 포함됨

-운영체제나 응용 프로그램이 파일을 올바르게 해석하고 처리하기 위한 ‘설명서’ 역할

-4D 5A는 DOS 파일에 대한 헤더로, 모든 실행 파일이 동일한 값 가짐 But, 78 00 의 경우 환경이나 프로그램마다 다를 수 있는 값.

→ Windows의 실행 파일의 앞 4바이트가 항상 4D 5A 78 00 인 건 X

### (5) Assembly(어셈블리어)

-nop 그리고 mov rax, 0x1302 와 같은 형태의 명령어 문구

-기계어를 사람이 보다 쉽게 이해할 수 있게 함

-어셈블리어로 컴퓨터가 수행할 명령을 작성한 뒤 기계어 즉, 프로그램으로 변환할 수도 있는 프로그래밍 언어

기계어: 컴퓨터가 실행해야 할 명령어

ex) 90(16진수)=nop(명령어)

nop → 아무 작업을 하지 않음

mov rax, 0x1302 → rax 라고 불리는 저장 공간에  0x1302 값을 저장함

**기계어와 그에 대응하는 어셈블리어**

| 기계어 | 어셈블리어 |
| --- | --- |
| 48 c7 c0 02 13 00 00 | mov rax, 0x1302 |
| 48 83 c3 05 | add rbx, 5 |
| 0f 84 ae 00 00 00 | je 0xb4 |

간단 정리

**프로그램에는 기계어가 포함되어 있고, 이 기계어는 실제 컴퓨터가 수행할 동작을 명령하는 것**

## 2) 프로그램 만들어 보기

**목표**

- 프로그램이 어떻게 만들어지는지 알아봄 → 리버스 엔지니어링의 기초 다지기(개발자가 작성한 소스 코드가 컴파일 과정을 거쳐 실행 가능한 바이너리 파일로 변환되는 일련의 과정 살펴볼 것)

### (1) 소스 코드 작성하기

**소스 코드란?**

-사람이 읽을 수 있는 프로그래밍 언어로 작성된 코드

-메모장 같은 문서 편집기를 이용하여 만들어짐(ex: 리눅스 대표 문서 편집기는 vim)

**작성 예시**

1. 리눅스에서 터미널 열기
2. vi helloworld.c 입력 → helloworld.c 라는 이름으로 vim 편집기를 연 후 쓸 내용 준비
3. 새로 만든 파일이므로 아무런 내용 X → i를 눌러 편집 모드로 들어가기(성공 시 좌하단에 INSERT라는 문자열 보임)
4. 소스 코드 작성
5. ESC 키를 눌러 편집 모드에서 빠져나오고, :wq 를 입력하여 파일 저장 후 vim 종료
6. 정상적으로 작성된 파일일 시 cat 명령어를 통해 내용 확인됨

### (2) 컴파일하여 프로그램 만들기

-프로그래밍 언어로 작성된 소스 코드와 프로그램은 별개의 파일

**Compile(컴파일) 이란?**

-사람과 컴퓨터 사이의 번역기

-사람이 작성한 C 수준의 고급 언어를 컴퓨터가 이해할 수 있는 기계어로 변환하고, 프로그램으로 만드는 과정

**위 소스 코드를 컴파일하여 프로그램으로 만들어 보기**

```c
gcc helloworld.c -o helloworld
```

를 입력하여 작성한 소스 코드를 컴파일

- gcc: C언어로 작성된 소스 코드 → 프로그램 으로 변환해 주는 컴파일러
- 위 명령어를 실행하게 되면 소스 코드가 컴파일 되어 **-o** 뒤에 작성한 **helloworld** 라는 이름으로 프로그램이 생성됨
- 컴파일 완료 시 프로그램이 생성됨 → 터미널에서 ./helloworld 입력하면 컴파일 된 프로그램 실행 가능

### (3) 프로그램 살펴보기

-Windows에서는 메모장으로 실행 파일을 열어 보았다면, Linux에서는 cat 명령어를 이용하여 파일 내용 확인 가능

- Windows에서 `.exe` 파일을 메모장으로 열면 깨지는 것처럼,
Linux 실행 파일도 `cat`으로 보면 이상한 문자들이 출력됨.

예시:

```
cat helloworld
```

→ 사람이 읽기 어려운 문자들이 막 출력됨.

-Windows에서는 HxD 편집기를 사용했다면, Linux에서는 xxd가 비슷한 역할 수행

실행 파일 구조를 더 보기 쉽게 확인하려고 `xxd` 사용함.

```
xxd helloworld
```

→ 파일 내용을 16진수(hex) 형태로 보여 줌.

내용이 너무 길어서:

```
xxd helloworld | head-3
```

→ 앞의 3줄만 출력.

여기서 중요한 부분:

```
7f 45 4c 46
```

- `45 4c 46`를 ASCII로 바꾸면 `E L F`

→ 리눅스 실행 파일은 `ELF` 형식 사용

ELF 뜻: Executable and Linkable Format

즉,

- Windows 실행 파일 → `.exe`
- Linux 실행 파일 → `ELF`

#### 리눅스에서 사용하는 실행 파일은 \x7fELF 로 시작

## 3) 동적 분석과 정적 분석 알아보기

**목표**

-본격적으로 리버스 엔지니어링 학습

**Reverse Engineering(리버스 엔지니어링) 이란?**

- 엔지니어링 과정을 역으로 수행하며 제품이 동작하는 원리와 제품에 들어간 기술들을 파악하는 것
- 역공학 or 리버싱이라고 칭하기도 함
- 프로그램을 분석하는 행위를 아울러 리버싱이라고 부름

#### 왜 해킹 분야로 분류되나?

프로그램 분석 자체가 이후 공격의 기반이 되기 때문

1. **프로그램 기술 탈취 가능**
- 실행 파일을 분석해 소스 코드나 알고리즘을 알아낼 수 있음
- 경쟁사 기술 복제 등에 악용 가능
ex) 게임 엔진 분석, 상용 프로그램 내부 로직 복제
2. **취약점 분석에 사용됨**
-시스템 해킹 전 보통 프로그램 구조부터 분석
ex) 버퍼 오버플로우 찾기, 인증 우회 로직 찾기, 숨겨진 함수 찾기
즉, 어디를 공격해야 하는지 알아내는 과정
3. **프로그램 변조 가능
-** 리버싱 후 프로그램을 수정해 원래 제한을 우회할 수 있음
ex) 게임 핵 제작, 라이선스 우회(crack)

**동적 분석과 정적 분석**

리버스 엔지니어링에서는 프로그램의 동작을 알아내기 위해 사용하는 분석 방법들을 크게 정적 분석과 동적 분석으로 구분

정적 분석: 외적인 관찰만을 통해 정보를 알아냄

동적 분석: 실행을 통해 동작을 분석함

- 정적인 방법과 동적인 방법을 적절히 혼용해야 효과적 분석 가능
- 동적 only: 큰 구조 놓칠 수 있음
정적 only: 프로그램의 실행을 통해 직관적으로 알 수 있는 부분을 분석하느라 시간 허비

### (1) 동적 분석 수행하기

#### 프로그램을 분석해 패스워드 알아내기

- **올바른 패스워드 길이 찾기**
1. 리눅스 터미널에서 프로그램이 존재하는 경로에서 ./main 실행 → 아래와 같이 출력

1 $ ./main

2 password:

1. 패스워드에 입력 넣어 보기

AAAA 입력 시 입력이 매우 짧다고 나옴

Your input is too long.

너무 길다고 출력됨

1. 여러 길이의 입력을 넣어 보며 올바른 길이의 입력 찾기

올바른 길이의 입력을 넣었을 때 나오는 출력

1   $ ./main
2   password: AAAAAAAA
3   Your input is not a number.

**(1) 8길이의 0 입력 시…**

```c
$ ./main
password: 00000000
output: 0xb9 0xb9 0xb9 0xb9 0xb9 0xb9 0xb9 0xb9

```

output 값=wanted 값이어야 함을 추측할 수 있음

**(2) 8길이의 1 입력 시…**

```c
$ ./main
password: 11111111
output: 0x62 0x62 0x62 0x62 0x62 0x62 0x62 0x62
wanted: 0x62 0xb1 0xb7 0xee 0xe8 0x65 0xee 0xe3
1/8 correct. Wrong password...
```

→ 패스워드의 첫 번째 값이 1일 것이라고 추측 가능
- 효율적 전략:
0~9까지의 숫자에 대한 output의 값이 어떻게 출력되는지 관찰.
wanted의 값과 0~9 숫자에 대한 output 값 중 일치하는 숫자 찾기

### (2) 정적 분석 수행하기

#### 정적 분석

프로그램을 실행하지 않고 분석하는 것

프로그램을 실행해도

- 모든 코드가 실행된다는 보장 X
- 특정 환경에서만 동작할 수 있음
- 악성코드일 위험성

→ 실행만으로는 분석 한계 존재

**그럼 뭘 분석하나?**

- 기계어(machine code)
- 바이너리 코드

But 기계어는 사람이 읽기 어려움

**→ 해결 방법: 디컴파일러(Decompiler)**

디컴파일러가

기계어 → C 수준의 프로그래밍 언어

변환해 줌

#### Ghidra 디컴파일러

1. **프로젝트 생성**
- 새 프로젝트 생성 → Non-Shared Project → 경로 원하는 곳으로 설정
1. **분석할 파일 추가**
- 드래그-앤-드랍으로 분석 파일 대상을 Ghidra 창에 끌어다 놓으면 프로젝트에 대상 등록됨
